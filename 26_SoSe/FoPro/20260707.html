Simulation von Alphafehler-Raten bei Bayes-Modellen

Einführung

In Bayes-Modellen spielt der Alphafehler zumeist keine Rolle. Der Grund ist, dass Bayes nicht daran interessiert ist (primär). In Bayes geht es um die Wahrscheinlichkeitverteilung des untersuchten Paramaters (d.h. der Hypothese) gegeben der Daten. Im Frequentismus, wo der Alphafehler beheimatet ist, geht es nicht um die Wahrscheinlichkeit der Hypothesen. Sondern darum, auf Dauer die H0 nicht zu oft/selten zu verwerfen. (Ob die H0 überhaupt sinnvoll als falsch zu definieren ist, steht auf einem anderen Blatt.) Darum spielt (nur) im Frequentismus der Alphafehler eine wichtige Rolle. Zur Erinnerung: Der Alphafehler sagt nicht, ob die vorliegende Hypothese richtig oder falsch ist. Sondern: wenn man den Versuch (unendlich) oft wiederholt unter Annahme der H0, wie oft man dann fälschlich die H0 verwirft. Kurz: Die Idee des Frequentismus ist, auf Dauer nicht zu viele Fehler zu machen. In Bayes ist die Idee, die Wahrscheinlichkeit der Hypothese im Licht der Daten und der Annahmen korrekt zu bewerten.

Aber man kann trotzdem in Bayes-Analyse prüfen, ob ein Entscheidungskriterium - z.B. theta ist außerhalb des 95%-Intervalls - auch wirklich die angezeigte Häufigkeit (wie 95%) trifft. Dann spricht man einer “Kalibrierung” des Modells; man untersucht die “Copverage” (Abdeckung) des Bayes-Modells. Das ist das Thema dieser Untersuchung.

Nullhyp testen

H0 wahr

Wenn die H0 wahr ist, dann wird sie in 95% richtig erkannt werden und in 5% der Fälle falsch erkannt werden

set.seed(1)
rate <- function(n, reps = 5000) {
  mean(replicate(reps, {
    x <- rnorm(n); y <- rnorm(n)   # delta = 0 exakt
    t.test(x, y)$p.value < 0.05
  }))
}
sapply(c(50, 500, 5000, 50000), rate)
[1] 0.0530 0.0520 0.0486 0.0548
#> ~0.05  0.05  0.05  0.05  -- konstant, n ist egal

H0 falsch, auch nur ein kleines, kleines bisschen

Wenn die H0 aber auch nur ein kleines bisschen falsch ist, wird - bei großem n - der Test immer signifikant werden.

set.seed(1)
rate <- function(n, reps = 5000) {
  mean(replicate(reps, {
    x <- rnorm(n); y <- rnorm(n, mean = 0.1)   # delta = 0.1, Mini-Effekt
    t.test(x, y)$p.value < 0.05
  }))
}
sapply(c(50, 500, 5000, 50000), rate)
[1] 0.0810 0.3512 0.9988 1.0000

Kalibrierung von Bayesmodellen testen

Geschwindigkeitstest für die Berechnung eines einzelnen Modells:

one_fit <- function(n, true_beta = 0.3, level = 0.95) {
  x <- rnorm(n)
  y <- true_beta * x + rnorm(n)
  fit <- stan_glm(y ~ x, data = data.frame(x, y),
                  chains = 2, iter = 1000, refresh = 0,
                  prior = normal(0, 2.5))
  ci <- posterior_interval(fit, prob = level, pars = "x")
  ci[1] <= true_beta && true_beta <= ci[2]
}

one_fit <- function(n, true_beta = 0.3, level = 0.95,

Funktion definieren, hier mit reps = 200.

check_coverage <- function(n, true_beta = 0.3, reps = 200, level = 0.95,
                           prior = normal(0, 2.5)) {
  hits <- vapply(seq_len(reps),
                 function(i) one_fit(n = n, true_beta = true_beta,
                                     level = level, prior = prior),
                 logical(1))
  mean(hits)
}

Problem - theta ist nicht fix

Der Effekt (theta) ist aber nicht fix - wir wissen ihn nicht genau. 100% der Wahrscheinlichkeit auf einen einzelnen Punkt zu setzen, ist nicht realistisch.

Wir können die Situation bessern und genauer schätzen, wenn wir mehrere Werte für theta prüfen.

betas <- c(0, 0.3, 0.6, 1)
# milder Prior (flacher Fall, alle ~0.95)

sapply(betas, function(b) check_coverage(n = 100, true_beta = b, reps = 200))

Im Überblick mit mehreren betas

run_grid <- function(prior, label, betas, n = 100, reps = 50) { # besser wäre: reps=1000, aber das dauert ein paar Stunden Rechenzeit
  cov <- vapply(betas,
                function(b) check_coverage(n = n, true_beta = b,
                                           reps = reps, prior = prior),
                numeric(1))
  data.frame(true_beta = betas, coverage = cov, prior = label, reps = reps)
}

Jetzt die betas übergeben:

betas <- c(0, 0.5, 1)
